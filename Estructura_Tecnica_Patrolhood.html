2. Lógica del Sistema SOS (Resiliencia Total)

  • 1
    Disparo y Buffer Local El cliente valida la integridad del mensaje. Si no hay conexión, la alerta se guarda en IndexedDB y se sincroniza automáticamente al recuperar señal.
  • 2
    Persistencia en MongoDB La señal SOS se registra en la colección ActiveSOS antes de cualquier otra acción. Esto garantiza la recuperación de alertas tras reinicios fortuitos del servidor.
  • 3
    Distribución Filtrada El servidor retransmite la alerta vía Socket.io únicamente a los miembros de la comunidad destino (validación por communityId).
  • 4
    Notificaciones Críticas Se ejecutan procesos asíncronos para el canal del foro, Firebase Cloud Messaging y el Bot de Telegram específico de la comunidad.

3. Seguridad de Nivel Empresarial

El sistema garantiza el aislamiento total de datos mediante una arquitectura multitenant basada en UUIDs.

Capas de Protección

  • Autenticación JWT: Cada petición API requiere un token firmado que certifica la role y el communityId del usuario.
  • Middleware Multi-tenant: El middleware checkCommunity intercepta todas las peticiones para asegurar que un usuario nunca acceda a recursos de otra comunidad.
  • Aislamiento de Identidad: Migración completa de nombres a communityId (UUID v4) para evitar colisiones y leaks de datos.
  • Auditoría (Audit Logs): Se registran todas las acciones administrativas críticas (borrado de mensajes, cambio de tokens, etc.) en una colección dedicada.
🛡️ Moderación Activa: Los administradores disponen de un Dashboard para visualizar logs de auditoría y la capacidad de borrar mensajes del foro en tiempo real mediante sockets.
